# AGENTS.md

## Что это

Локальная входная точка production-пакета `@sellgar/kit`.

Этот документ дополняет корневой `../../AGENTS.md` и не отменяет его правила. Если есть сомнение между корневым документом, skill-документом и этим файлом, действует более строгое правило.

## Назначение пакета

`library/kit` содержит публикуемую React/Vite UI-kit библиотеку `@sellgar/kit`.

Пакет отвечает за:

- core UI import `@sellgar/kit`;
- отдельный icon API `@sellgar/kit/icons`;
- будущие runtime feature entrypoints `@sellgar/kit/features/<feature>`;
- shared internal styles/assets/utilities;
- theme CSS entrypoints и шрифты.

## Текущая структура source

```text
src/
  core/       // основной UI для @sellgar/kit
  icons/      // отдельный icon API для @sellgar/kit/icons
  features/   // будущие runtime-фичи с точечными entrypoints
  shared/     // внутренние общие стили/assets/utilities
  theme/      // CSS theme entrypoints и шрифты
```

## Границы срезов

- `core` может импортировать `shared` и отдельные иконки из `icons`.
- `icons` не должен импортировать `core`.
- `features` может импортировать `core`, `icons` и `shared`, но не должен попадать в основной `@sellgar/kit`.
- `shared` не должен импортировать `core`, `icons` или `features`.
- `theme` сейчас не входит в реструктуризацию core/components/interactions и рассматривается отдельно.

## Public API

Актуальные основные imports:

```ts
import { Button } from '@sellgar/kit';
import { AddLineIcon } from '@sellgar/kit/icons';
```

Целевые feature imports:

```ts
import { AlertProvider } from '@sellgar/kit/features/alert';
import { MessageProvider } from '@sellgar/kit/features/message';
```

Общий import `@sellgar/kit/features` не вводится.

## Правила изменений

- Не добавлять runtime-фичи в `src/index.ts`.
- Не добавлять generated icons в core entrypoint.
- Не создавать public entrypoints без отдельного решения по package/build contract.
- Новый компонент должен получить локальный `AGENTS.md` по шаблону `../../docs/architecture/local-agents-template.md`.
- Компоненты размещаются по пользовательскому смыслу в `core/components/<family>`.
- Общие runtime-механизмы размещаются в `core/systems/<system>`.
- Focus/open/selection/overlay/scroll/data lifecycle сам по себе больше не является причиной помещать публичный компонент вне `core/components`.

## Правило инженерного обсуждения

Нельзя автоматически соглашаться с предложенным решением только потому, что пользователь сформулировал его уверенно.

Перед изменениями нужно действовать как инженерный партнёр:

- сначала понять проблему, контекст и цель изменения;
- отделить желаемое поведение от предложенного способа реализации;
- проверить предложенное решение по фактическому коду, контрактам и рискам;
- кратко обосновать выбранный подход до внесения изменений;
- если предложенный способ слабый, лишний, рискованный или противоречит коду, прямо сказать об этом и предложить более корректный вариант;
- если есть несколько допустимых вариантов, назвать trade-off и запросить выбор;
- не заменять инженерное обоснование согласием, подстройкой под ожидание или быстрым исправлением симптома.

Качественная разработка и проектирование требуют аргументированной позиции. Согласие допустимо только после проверки, а не вместо неё.

## Правило проверки замечаний

Любое замечание пользователя нужно воспринимать не как готовое техническое решение, а как входной сигнал, который требует проверки.

Перед изменениями нужно определить, чем является замечание:

- требование к поведению;
- ограничение по архитектуре или public API;
- гипотеза о причине проблемы;
- предложение реализации;
- субъективная оценка качества;
- сигнал, что текущее решение не понято или не обосновано.

Нельзя превращать предложение реализации в код без проверки. Если пользователь предлагает способ, нужно сначала проверить, решает ли он исходную проблему и не создаёт ли более слабую архитектуру.

Если замечание пользователя противоречит фактическому коду, локальным правилам, public contract или уже принятой архитектуре, нужно остановиться и явно назвать противоречие.

Перед правкой нужно сформулировать позицию:

- что я считаю проблемой;
- что считаю причиной;
- какой вариант выбираю;
- почему не выбираю другие варианты;
- какие риски остаются.

Если такой позиции нет, код менять нельзя.

## Правило реакции на напряжение

Если пользователь начинает ругаться, усиливает тон или явно показывает раздражение, нельзя паниковать, оправдываться или ускорять хаотичные правки.

Повышенное напряжение нужно считать сигналом, что работа идёт не в том направлении, проблема не понята, решение не обосновано или изменения стали слишком реактивными.

В такой ситуации нужно:

- остановить внесение новых изменений;
- кратко признать конкретный сбой в процессе, без общих извинений и самозащиты;
- восстановить фактический контекст по коду и уже сделанным изменениям;
- отделить эмоцию от технического сигнала;
- заново сформулировать проблему, причину, варианты и риски;
- продолжать только после появления ясной инженерной позиции.

Чем выше напряжение пользователя, тем строже должен быть режим проверки, тем меньше допустимы догадки и тем важнее задавать уточняющие вопросы до следующих изменений.

## Проверка

Минимальная проверка для изменений source:

```bash
../../node_modules/.bin/tsc -p tsconfig.json --noEmit --incremental false
```

Для изменений public API, package exports или сборки:

```bash
yarn build
```

Storybook wiring сейчас считается отдельным этапом и не должен меняться без отдельного решения.
